# Day 07 - Linux File System Hierarchy and Scenario-Based Practice

## Linux File System Hierarchy

### Let's understand where important files and directories live in Linux

---

## Core Directories (Must Know)

### `/` (Root)

- The root directory is the starting point of the Linux file system.
- Every file and directory exists under this location.

**Command Used:**

```bash
ls -l /
```

**I would use this when:**
I want to see the top-level structure of the Linux system.

---

### `/home`

- Contains home directories of normal users.
- User-specific files and folders are stored here.

**Command Used:**

```bash
ls -l /home
```

**I would use this when:**
I need to access or manage user files.

---

### `/root`

- Home directory of the root user.
- Used for administrative tasks.

**Command Used:**

```bash
sudo ls -l /root
```

**I would use this when:**
I need to work with files owned by the root user.

---

### `/etc`

- Stores configuration files for the operating system and services.
- Most service settings are managed from this directory.

**Command Used:**

```bash
ls -l /etc
cat /etc/hostname
```

**I would use this when:**
I need to view or modify system configurations.

---

### `/var/log`

- Stores log files generated by the system and applications.
- Very important for troubleshooting.

**Command Used:**

```bash
ls -l /var/log
```

**I would use this when:**
I need to investigate errors or service failures.

---

### `/tmp`

- Temporary files are stored here.
- Files may be removed automatically after reboot.

**Command Used:**

```bash
ls -l /tmp
```

**I would use this when:**
Applications create temporary working files.

---

## Additional Directories (Good to Know)

### `/bin`

- Contains essential Linux commands.
- Commands such as `ls`, `cp`, `mv`, and `cat` are available from here.

**I would use this when:**
I want to know where important command binaries are stored.

---

### `/usr/bin`

- Contains most user-level command binaries.
- Many commonly used programs are located here.

**I would use this when:**
I need to locate installed command-line tools.

---

### `/opt`

- Used for optional or third-party software installations.
- Applications not included with the operating system are often installed here.

**I would use this when:**
I want to check manually installed software.

---

# Scenario-Based Practice

## Scenario 1: Service Not Starting

### Problem

A service named `myapp` failed to start after a server reboot.

### Step 1

```bash
systemctl status myapp
```

**Why:**
Check whether the service is active, inactive, or failed.

### Step 2

```bash
journalctl -u myapp -n 50
```

**Why:**
Review recent logs to identify the cause of failure.

### Step 3

```bash
systemctl is-enabled myapp
```

**Why:**
Verify whether the service is configured to start automatically during boot.

### Step 4

```bash
systemctl restart myapp
```

**Why:**
Attempt to restart the service after reviewing logs.

### What I Learned

Always check service status first, then investigate logs before making changes.

---

## Scenario 2: High CPU Usage

### Problem

The application server is responding slowly and CPU utilization is very high.

### Step 1

```bash
top
```

**Why:**
Monitor live CPU and memory usage.

### Step 2

```bash
ps aux --sort=-%cpu | head -10
```

**Why:**
Identify processes consuming the highest CPU resources.

### Step 3

```bash
pidof nginx
```

**Why:**
Find the process ID of a specific application.

### Step 4

```bash
top -p <PID>
```

**Why:**
Monitor resource consumption of a specific process.

### What I Learned

CPU bottlenecks can often be identified by checking the top resource-consuming processes.

---

## Scenario 3: Finding Service Logs

### Problem

A developer wants to check Docker service logs.

### Step 1

```bash
systemctl status docker
```

**Why:**
Verify the current status of the Docker service.

### Step 2

```bash
journalctl -u docker -n 50
```

**Why:**
View the latest Docker service logs.

### Step 3

```bash
journalctl -u docker -f
```

**Why:**
Follow Docker logs in real time.

### Step 4

```bash
docker logs <container_id>
```

**Why:**
View logs of a specific running container.

### What I Learned

Systemd-managed services store logs in the journal and can be accessed using `journalctl`.

---

## Scenario 4: Permission Denied Error

### Problem

A script named `backup.sh` shows "Permission denied" when executed.

### Step 1

```bash
ls -l backup.sh
```

**Why:**
Check the current file permissions.

### Step 2

```bash
chmod +x backup.sh
```

**Why:**
Add execute permission to the script.

### Step 3

```bash
ls -l backup.sh
```

**Why:**
Verify that execute permission has been added.

### Step 4

```bash
./backup.sh
```

**Why:**
Run the script after correcting permissions.

### What I Learned

Linux permissions control who can read, write, and execute files.

---

# Key Takeaways

- Linux follows a structured file system hierarchy.
- `/etc` stores configuration files.
- `/var/log` is essential for troubleshooting.
- `systemctl` is used to manage services.
- `journalctl` helps investigate service logs.
- File permissions are important for security and script execution.
